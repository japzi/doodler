## Problem Statement

Doodler v0.2 assumed every scene object is a `<path>` with a `pathData` string. v0.3 needed a text tool that breaks this assumption: text renders as SVG `<text>` elements, has no path data, and requires an inline editing UX (click to place, type, commit). The font had to match the hand-drawn aesthetic (Humor Sans), and SVG export needed to embed the font so pasted text renders correctly outside the app.

## Solution Implementation

**Text as a distinct object type.** `TextObject` has `text`, `fontSize`, `color`, `position`, and `boundingBox` — but no `pathData`. SceneRenderer was split into two memo'd sub-components (`TextElement` and `PathElement`) with a conditional branch on `obj.type === 'text'`.

**Click-to-place textarea overlay.** The text tool hook sets `activeTextInput` coordinates on click, which causes `TextInputOverlay` to render a `position: fixed` HTML `<textarea>` at the computed screen position. The textarea sits outside the SVG as a sibling element, styled with Humor Sans at the viewport-scaled font size. Commit happens on Escape or blur — creating a new `TextObject` (or updating an existing one on re-edit, or deleting if emptied).

**Double-click re-editing.** A `handleDoubleClick` on the SVG hit-tests for `data-object-id`, and if the target is a text object, sets `editingTextId`. The SceneRenderer hides the object being edited so it doesn't show behind the textarea. The textarea pre-fills with the existing text.

**Async SVG export with embedded font.** `serializeToSvg` became async. When any text objects exist, it fetches `/fonts/HumorSans.ttf`, base64-encodes it, and injects an `@font-face` `<style>` block into the SVG. The base64 result is cached at module level so subsequent exports skip the fetch. Text objects serialize as `<text font-family="'Humor Sans', cursive">` with `<tspan>` elements for multi-line support.

**Multi-line support.** Text with `\n` characters renders as multiple `<tspan>` elements with `dy` offsets based on `fontSize * 1.3` line height. The `measureText` utility splits on newlines and takes the max line width for the bounding box.

## Development Learnings

**Textarea focus timing with SVG pointer events.** The initial implementation focused the textarea synchronously in a `useEffect`, but the user reported "when I try to type text it doesn't render." The root cause was a focus/blur race: the browser's pointer event processing on the SVG could cause the textarea to blur immediately after focusing, committing empty text before the user typed anything. Three fixes were applied:

1. `e.preventDefault()` in the text tool's `onPointerDown` to prevent browser default focus behavior during the click
2. `requestAnimationFrame` wrapper around `textarea.focus()` so focus happens after the pointer event fully processes
3. A `committedRef` guard to prevent double-commits (Escape triggers commit, then textarea unmount triggers blur which would commit again)
4. `setTimeout(0)` on the blur handler to defer commit until after any focus handoff completes
